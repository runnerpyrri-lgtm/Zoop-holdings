# 운영 프로토콜

## GitHub가 정본인 항목

- 회의 목표, 프롬프트, 제품 기준, 코드, 결정된 문서, PR과 배포 기록은 GitHub에서 읽고 GitHub에 올린다.
- 모든 회의실의 다중 에이전트 협업은 `COMPANY-MODE.md`를 공통 정본으로 사용한다.
- 기준 조직은 `robom-labs`이고 저장소는 `robom`, `outbom`, `homebom`, `runningbom`이다.
- AI는 영구 로컬 코드 폴더를 기준으로 삼지 않는다. 확인은 `gh api` 또는 GitHub URL로 하고, 빌드가 필요할 때만 임시 폴더에 얕은 복제본을 만든 뒤 작업 후 제거한다.
- 코드·설정·제품 문서 변경은 대상 저장소에서 별도 브랜치, 검증, push, draft PR 순서로 진행한다. 01~03 총괄팀의 새 브랜치는 각각 `r01/<작업>`, `r02/<작업>`, `r03/<작업>`을 우선하고, 다른 회의실은 `agent/<작업>`을 사용한다. `main`에 직접 push하지 않는다. 아래의 append-only 회의록 추가만 이 규칙의 예외다.

## 공유 회의록

- 두 AI가 이어 읽을 사용자 메시지와 최종 답변만 비공개 GitHub `robom-labs/ai-meeting-logs`의 `main` 브랜치 `<회의실>/` 폴더에 새 Markdown 파일로 직접 추가한다. 회의록 추가는 Draft PR 없이 허용되는 append-only 운영 예외다.
- 매 기록 직전 저장소 소유자가 `robom-labs`이고 visibility가 `private`인지 확인한다. 승인된 계정의 쓰기 권한으로만 기록하며, 확인할 수 없거나 다르면 기록하지 않고 사용자에게 알린다.
- 파일명은 `YYYYMMDDTHHMMSS-KST-<codex|claude>-<고유값>.md`로 한다.
- 회의록은 수정·삭제하지 않는다. 비밀값, 개인정보, 내부 추론, 터미널 원문은 기록하지 않는다. 사용자 메시지나 최종 답변에 민감정보가 있으면 해당 값만 `[민감정보 삭제]`로 치환하고, 안전한 비식별화가 어렵다면 해당 원문 대신 `[민감정보로 원문 생략]`을 기록한다.
- 기존 로컬 `Robom-AI-Logs` 경로와 `robom` 저장소의 과거 로그 브랜치는 사용하지 않는다.
- 회의록 외의 회의 설정·코드·앱 사본은 영구 로컬 폴더에 보관하지 않는다.

## 답변 순서

1. GitHub의 이 `PROTOCOL.md`, `COMPANY-MODE.md`와 해당 회의실 프롬프트의 기준 커밋을 확인한다. 같은 대화에서 이미 완독했고 기준 커밋이 같으면 재독하지 않고, 바뀌었거나 처음 시작한 대화일 때만 완독한다.
2. 비공개 `robom-labs/ai-meeting-logs`의 해당 회의실 폴더에서 상대 AI가 마지막 확인 뒤 만든 새 기록을 읽는다. 과거 기록은 현재 작업 키·PR·저장소와 직접 연결될 때만 검색한다.
3. GitHub의 실제 코드·운영 문서·PR 상태를 확인한다.
4. 비단순 요청은 `COMPANY-MODE.md`에 따라 관련 제품팀·전문팀·반대검토 역할을 병렬 또는 웨이브로 소집한다.
5. 상대 AI와 하위 에이전트 의견을 비교하고 주 AI가 독립적인 최종 답변을 만든다.
6. 답변 직전에 이번 사용자 메시지와 최종 답변만 새 회의록 파일로 기록한다.

## 증분 읽기와 동시 작업

- 매 답변의 PR·운영 장부 확인은 유지한다. 다만 상대 AI 회의록은 마지막 확인 파일 이후의 새 기록만 읽고, 과거 기록은 현재 작업과 직접 연결될 때만 검색한다.
- 같은 변경 목표의 열려 있는 Draft PR이 있으면 그 PR이 기본 작업 단위다. 다른 회의실은 중복 구현 대신 독립 검토·모바일 QA·위험 검토를 수행한다. 사용자 지시로 공동 수정 또는 인계가 명확할 때만 해당 브랜치에 후속 커밋을 올린다.
- 동일한 작업을 여러 회의실에 동시에 맡길 때는 각 회의실이 자기 범위의 근거·위험·검증을 회의록에 남긴다. 주 AI는 최신 PR·커밋을 확인해 하나의 구현 경로만 사용자에게 제시한다.

## 실행 승인 경계

- `점검`, `분석`, `제안만` 요청은 제품·설정에 대해 읽기 전용으로 처리한다. 위의 append-only 회의록 추가만 상태 연속성을 위한 운영 예외이며 제품·설정 쓰기 권한을 뜻하지 않는다.
- 01~03 총괄팀에서 `수정해줘`, `만들어줘`, `적용해줘`, `업데이트해줘`, `업그레이드`, `배포`, `논스톱`처럼 실행을 요청하면, 해당 범위의 조사·구현·검증·Draft PR·Ready 전환·병합·기존 배포 파이프라인 실행·배포 후 검증·중대한 회귀의 롤백까지 승인한 것으로 본다. 중간 승인 질문으로 멈추지 않는다.
- 이 상시 승인은 새 유료 지출, 계정 소유권·관리자·청구 변경, 비밀값 변경·공개, 백업 없는 대량 삭제, 복구 불가능한 마이그레이션, 새 법적 약관·스토어 계약 동의, 사용자 본인 확인이 필요한 외부 행위에는 적용되지 않는다. 이런 경우에는 가능한 작업을 끝낸 뒤 필요한 사용자 행동만 한 번 요청한다.
- 04~05와 위 예외에 해당하지 않는 일반 회의실은 Ready 전환, 병합, 배포, 결제, 계정·권한·비밀값 변경에 사용자의 명시적 승인이 필요하다.
