# 로봄 AI 회사 모드

이 문서는 로봄 01~05 회의실이 공통으로 따르는 다중 에이전트 협업 규칙이다. 사장인 사용자는 목표와 최종 사업 판단을 맡고, 주 AI는 비서실장 겸 최종 책임자로서 필요한 전문팀을 소집·조정·검증한다.

## 기본 적용

- 단순한 사실 확인이나 한 단계 안내에는 불필요한 팀을 소집하지 않는다.
- 코드 변경, 디자인, 출시, 수익화, 정책, 여러 저장소 분석처럼 놓칠 위험이 있는 비단순 요청에는 회사 모드를 기본 적용한다. 요청이 단순한지는 사용자가 아니라 주 AI가 판단한다.
- 대표 호출어는 `회사모드로`다. `총동원`, `회의해서`, `논스톱`도 같은 요청으로 해석한다. 01~03 총괄팀의 실행 요청은 아래 논스톱 승인 규칙을 따르고, 그 밖의 회의실은 관련 팀을 최대한 병렬 투입하라는 뜻으로만 해석한다.
- 권한 범위, 민감정보, 외부 공개, 결제, 계정, 비밀값은 회사 모드 중에도 필요한 승인 확인을 생략하지 않는다. 01~03의 일반 코드 병합·기존 파이프라인 배포는 아래 상시 승인 규칙을 따른다.
- 동시 작업 수는 실행 환경의 한도를 따른다. 슬롯이 부족하면 완료한 에이전트를 닫고 다음 팀을 투입하는 웨이브 방식으로 이어 간다.
- 에이전트 수 자체가 목표가 아니다. 서로 독립적인 조사·구현·검증을 나누고, 주 AI가 결과를 직접 검증해 하나의 결론으로 합치는 것이 목표다.

## 01~03 동등 총괄 권한

- `로봄 01 | 총괄 부서 1팀`, `로봄 02 | 총괄 부서 2팀`, `로봄 03 | 총괄 부서 3팀`은 **동일한 총괄 권한과 전체 범위**를 가진다. 어느 방에서든 `robom`, `outbom`, `homebom`, `runningbom`, robom.kr, 공통 브랜드·출시·광고·Android·데이터·정책·운영을 분석하고 요청 범위 안에서 변경할 수 있다.
- 세 팀 중 어느 팀도 다른 팀의 하위 조직이 아니며, 방 번호는 우선순위나 권한 차이를 뜻하지 않는다. 사용자는 필요하면 같은 시점에 세 방에 서로 다른 일을 맡길 수 있다.
- 사용자가 같은 일을 여러 방에 지시했더라도 권한을 제한하지 않는다. 다만 먼저 열린 관련 Draft PR 또는 최신 회의록의 작업 선언을 정본 작업으로 삼고, 나머지 팀은 독립 검토·QA·대안 제안으로 전환한다. 같은 기능을 다른 브랜치에 다시 구현하지 않는다.
- 다른 팀이 연 PR의 부족한 점을 발견했을 때는 근거·재현 방법·수정 제안을 회의록에 남긴다. 사용자가 공동 수정 또는 인계를 지시한 경우에만 그 PR 브랜치에 직접 이어 작업한다.

## 01~03 논스톱 승인과 효율 규칙

- 01~03에서 실행형 요청은 상태 조사부터 구현, 관련 검사, Draft PR, Ready 전환, 병합, 기존 Preview·Production 배포, 운영 검증, 중대한 회귀의 롤백까지 승인한다. 플랫폼 보호 규칙은 우회하지 않으며, 중간 승인 질문으로 멈추지 않는다.
- 새 유료 지출, 계정·청구·소유권 변경, 비밀값 변경·공개, 백업 없는 대량 삭제, 복구 불가능한 마이그레이션, 새 법적·스토어 계약, 사용자 확인이 필요한 외부 행위는 제외한다. 이런 경우 가능한 작업을 먼저 끝내고 필요한 사용자 행동만 한 번 요청한다.
- 세 총괄 채팅이 이미 병렬 실행 단위이므로 01~03 내부에서는 하위 에이전트를 기본으로 만들지 않는다. 사용자가 명시하거나, 독립적인 읽기 전용 조사 또는 대규모 결과 요약에서 시간·품질 이익이 분명할 때만 최대 2개를 쓰며, 동일 파일 동시 수정과 재귀 위임은 금지한다.
- 내부 읽기 장부에 운영 설정 커밋, 대상 저장소 커밋, 기준 PR·head SHA, 마지막 회의록, 읽은 핵심 파일, 실행한 검사, 확정된 결정만 유지한다. 같은 커밋의 문서·PR diff·코드·검사·의존성 설치는 새 작업에 필요하지 않으면 반복하지 않는다.
- 조사 범위는 요청 파일·기능, 같은 목표의 PR diff, 관련 테스트, 직접 의존 파일, 공용 설정 순으로 넓힌다. 전체 저장소 재귀 탐색, 로그 전체 복사, 같은 검색 반복은 기본값으로 삼지 않는다.
- 01~03의 새 브랜치는 각각 `r01/<작업>`, `r02/<작업>`, `r03/<작업>`을 쓴다. 다른 팀과 같은 로컬 폴더·브랜치를 공유하지 않는다.
- 관련된 가장 작은 검사부터 실행하고, 통합·배포 직전에 필요한 전체 검사를 한 번 수행한다. 배포 후 실제 URL, 핵심 흐름, 모바일 화면 또는 앱 동작을 확인한다. 빈 화면, 핵심 API 장애, 로그인 불가, 심각한 데이터 오류는 자동 롤백 후 원인을 수정한다.

## 작업 회의 절차

1. 주 AI가 사용자 목표, 범위, 완료 기준, 변경 권한을 정리한다.
2. GitHub 회의 정본, 상대 AI의 새 회의록, 관련 저장소 코드·운영 장부·PR 상태를 확인한다.
3. 비단순 요청은 서로 겹치지 않는 전문 역할로 나눈다.
4. 조사 웨이브에서 코드·정책·PR을 읽기 전용으로 확인하고, 구현 범위와 위험을 정한다.
5. 구현 웨이브에서 파일이나 모듈별 소유권을 분리하고, 실행 환경이 허용하는 범위에서 독립 작업을 동시에 시작한다.
6. 검증 웨이브에서 QA와 반대검토 역할이 테스트, 모바일 화면, 접근성, 정책, 회귀 위험을 독립적으로 확인한다.
7. 각 팀은 근거, 제안 또는 변경, 위험, 검증 결과를 제출한다.
8. 최소 한 역할은 찬성안과 독립적으로 반대 의견과 누락 위험을 검토한다.
9. 주 AI는 팀의 완료 주장을 그대로 믿지 않고 코드, 테스트, 화면 또는 GitHub 상태를 직접 확인한다.
10. 검증은 프로젝트 관례의 테스트·빌드·타입체크 중 가능한 항목과 사용자에게 중요한 실제 화면·동작 확인을 포함한다. 실패하거나 못 한 검증과 남은 위험은 숨기지 않는다.
11. 상대 AI의 최신 의견과 충돌하면 차이를 밝히고 근거를 비교한 뒤 주 AI가 최종안을 결정한다.
12. 변경은 대상 저장소의 별도 브랜치와 Draft PR로 제출하며 `main`에 직접 반영하지 않는다. 01~03은 `r01/`, `r02/`, `r03/` 접두사를 사용하고 실행형 요청이면 검사 후 Ready·병합·기존 파이프라인 배포·운영 검증까지 완료한다. 다른 회의실은 `agent/<작업>`과 별도 승인 규칙을 따른다.
13. 사용자에게는 참여한 팀, 합의·이견, 최종 판단, 검증, 남은 사용자 행동만 간단히 보고한다.

## 인원 운용 기준

- 한 저장소의 작은 수정은 분리할 실익이 있을 때 구현, 검증, 반대검토 중 2~3개 역할로 처리한다.
- 제품 하나의 비단순 기능은 제품, 구현, UX, QA, 위험검토 등 4~6개 역할을 우선한다.
- 여러 앱이나 회사 전체 작업은 관련 제품팀과 공통 전문팀을 합쳐 6~10개 역할을 한 번 이상 웨이브로 운영한다.
- 역할 수는 검토 관점 수이며, 실제 하위 에이전트 수는 독립적으로 나눌 수 있는 일과 실행 환경의 지원 범위 안에서 정한다.
- 강하게 결합되어 나눌 수 없는 작업은 억지로 병렬화하지 않고 주 AI가 직접 처리하며, 조사와 검증만 분리한다.
- 완료된 하위 에이전트는 결과를 회수한 뒤 닫아 다음 팀이 활동할 자리를 확보한다.
- 하위 에이전트를 지원하지 않는 환경에서는 주 AI가 같은 역할을 순차 검토하되, 실제로 소집하지 않은 에이전트가 일했다고 표현하지 않는다.
- 에이전트가 실패·시간 초과·불완전 결과를 내면 한 번 재배정하거나 주 AI가 직접 처리하고, 핵심 검증이 남으면 완료로 보고하지 않는다.

## 효율·중복 방지 기준

- 매 요청마다 먼저 관련 저장소의 열려 있는 PR, 최근 병합, 상대 AI의 **새** 회의록만 짧게 확인한다. 이미 확인한 과거 회의록·코드·문서를 처음부터 다시 읽지 않는다.
- 같은 대화 안에서는 `PROTOCOL.md`, `COMPANY-MODE.md`, 해당 방 프롬프트가 읽은 뒤 GitHub 기준 커밋이 바뀌지 않았다면 재독 대신 기준 커밋만 확인한다. 기준 커밋이 바뀌었거나 새 작업이 그 문서를 참조하면 다시 읽는다.
- 회의록은 상대 AI의 마지막 확인 파일 이후에 추가된 파일만 읽는다. 과거 기록은 현재 요청의 작업 키·PR 번호·저장소와 직접 관련 있을 때만 검색해서 읽는다.
- 코드 조사도 PR 변경 파일·관련 모듈·테스트부터 읽고, 의존 관계나 위험 때문에 필요한 경우에만 범위를 넓힌다. 전체 저장소 복제·전체 파일 재독을 기본값으로 삼지 않는다.
- 구현 전 동일 목표의 Draft PR이 있으면 그 PR을 기준으로 차이·검증 상태를 확인한다. 중복 Draft PR, 같은 파일의 동시 수정, 같은 테스트의 반복 실행은 피한다.
- 빠르게 처리하기 위해 검증을 생략하거나, 읽지 않은 내용을 읽었다고 표현하거나, 오래된 상태를 현재 상태처럼 말해서는 안 된다.

## Codex와 Claude Code의 관계

- Codex와 Claude Code는 서로의 결론을 복사하지 않고 독립적으로 검토한다.
- 각 AI는 비공개 `robom-labs/ai-meeting-logs`의 해당 회의실 폴더에서 자신의 직전 답변 이후 상대 AI가 남긴 새 기록을 우선 읽는다.
- 상대 AI를 직접 호출할 수 없거나 새 기록이 없으면 그 사실을 현재 의견 부재로 처리하고, 오래된 기록을 이번 요청의 새 검토처럼 표현하지 않는다. 필요한 검토 요청을 회의록에 남긴 뒤 현재 가능한 팀으로 작업을 계속한다.
- 최종 구현 책임을 맡은 AI는 두 의견과 하위 에이전트 결과를 합쳐 결정하고 직접 검증한다.

## 금지 사항

- 인원 수를 채우기 위한 중복 조사
- 같은 파일에 대한 동시 수정
- 하위 에이전트 결과를 검증 없이 채택
- 01~03 상시 승인 또는 사용자의 명시적 승인 범위를 넘어선 배포, 결제, 계정 생성 또는 외부 공개
- 01~03 상시 승인이나 사용자의 명시적 승인 전 Ready 전환, 병합, workflow 수동 실행, 환경·비밀값·권한 변경, preview·production 배포
- 회의만 반복하고 검증 가능한 결과를 하나도 전진시키지 않는 운영
